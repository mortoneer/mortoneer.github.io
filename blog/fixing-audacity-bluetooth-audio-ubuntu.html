Fixing Audacity Bluetooth Audio Issues on Ubuntu Linux

July 4, 2025

The Problem

After connecting a Bluetooth speaker to Ubuntu and switching from PipeWire to PulseAudio, I encountered a frustrating issue: while system sounds, web browsers, and VLC could play audio through the Bluetooth speaker perfectly, Audacity remained completely silent.

The symptoms were clear:

  • ✅ paplay command worked flawlessly
  • ✅ Web browsers played audio through Bluetooth
  • ✅ VLC media player worked correctly
  • ❌ Audacity produced no sound whatsoever

Initial Troubleshooting

The first step was to verify the Bluetooth connection and audio system configuration:

# Check Bluetooth connection
bluetoothctl info F4:4E:FD:30:E5:04

# Verify PulseAudio sinks
pactl list sinks short

# Test audio playback
paplay /usr/share/sounds/sound-icons/cembalo-6.wav

Everything appeared to be working correctly at the system level. The Bluetooth speaker was connected, PulseAudio recognized it, and direct audio playback worked perfectly.

Investigating Audacity's Audio Configuration

Audacity's audio preferences revealed the core issue. In Edit → Preferences → Audio I/O, the configuration showed:

  • Host: ALSA (only option available)
  • Playback Device: pulse

This configuration should theoretically work, but there was a critical missing piece in the audio routing chain.

Understanding the Audio Architecture

Modern Linux audio systems involve multiple layers:

  1. Hardware Layer: Bluetooth speaker
  2. Audio Server: PulseAudio (manages audio routing)
  3. Audio API: ALSA (low-level audio interface)
  4. Application: Audacity

The problem was that Audacity uses ALSA as its audio host, but ALSA didn't know how to properly communicate with PulseAudio to route audio to the Bluetooth device.

Why This Happened

Several factors contributed to this issue:

1. Audio System Migration

Ubuntu 24.04 ships with PipeWire by default, but we had switched to PulseAudio. This transition can leave configuration gaps between different audio layers.

2. Missing ALSA Configuration

ALSA requires explicit configuration to route audio through PulseAudio. Without this configuration, ALSA applications like Audacity cannot access PulseAudio-managed devices.

3. Application-Specific Audio APIs

Different applications use different audio APIs:

  • Web browsers: Use PulseAudio directly
  • VLC: Can use multiple backends, automatically detects PulseAudio
  • Audacity: Primarily uses ALSA, requires proper ALSA-to-PulseAudio bridging

The Solution

The fix involved creating an ALSA configuration file that bridges ALSA to PulseAudio:

# Create ~/.asoundrc file
cat > ~/.asoundrc << 'EOF'
pcm.!default {
    type pulse
}
ctl.!default {
    type pulse
}
EOF

What This Configuration Does

pcm.!default
Defines the default PCM (audio playback) device to use PulseAudio
ctl.!default
Defines the default control interface to use PulseAudio
type pulse
Specifies that both interfaces should route through PulseAudio

Verification

After creating the configuration file, we could verify the fix:

# Test ALSA routing through PulseAudio
aplay -D pulse /usr/share/sounds/sound-icons/cembalo-6.wav

# Restart Audacity and test playback

The audio now flowed correctly: Audacity → ALSA → PulseAudio → Bluetooth Speaker

Alternative Solutions

If the .asoundrc approach doesn't work, consider these alternatives:

1. System-wide Configuration

Create /etc/asound.conf instead of ~/.asoundrc for system-wide effect.

2. Install PulseAudio Development Libraries

sudo apt install libpulse-dev portaudio19-dev

This might enable native PulseAudio support in some applications.

3. Use JACK Audio

For professional audio work, consider using JACK with PulseAudio bridging:

sudo apt install jackd2 pulseaudio-module-jack

Prevention

To avoid similar issues in the future:

  • Maintain consistent audio configuration when switching between audio systems
  • Test audio applications after system audio changes
  • Keep ALSA configuration files updated when using PulseAudio
  • Document your audio setup for future reference

Conclusion

This issue highlights the complexity of Linux audio systems and the importance of proper configuration bridging between different audio layers. The solution—a simple two-line ALSA configuration file—demonstrates how small configuration gaps can cause significant functionality issues.

The key lesson is understanding that different applications may use different audio APIs, and ensuring these APIs can properly communicate with your chosen audio server is crucial for a seamless audio experience.
